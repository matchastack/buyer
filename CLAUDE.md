# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Automated monitor and checkout assistant for Pokemon Center SG products on Lazada SG. TypeScript + Playwright (headed Chromium). Runs one worker per configured item that polls a product page, decides whether to buy, clicks Buy Now, selects PayNow, and places the order. Lazada then presents a PayNow QR which the user scans to complete payment.

First-time setup: copy `config.example.json` ‚Üí `config.json` and `.env.example` ‚Üí `.env`. Both are git-ignored. Credentials (`LAZADA_EMAIL`, `LAZADA_PASSWORD`) MUST come from env vars ‚Äî `config.ts` rejects any `credentials`/`email`/`password` key found in `config.json`. The same rule covers the optional proxy: `PROXY_USERNAME`/`PROXY_PASSWORD` are env-only (the non-secret `PROXY_SERVER`/`settings.proxy.server` may live in config).

### Anti-bot resilience (proxy + stealth)

Datacenter IPs are flagged by Lazada/Alibaba's anti-bot stack almost immediately under repeated polling, so the most effective mitigation is a **residential proxy**, configured via env vars (`PROXY_SERVER`, `PROXY_USERNAME`, `PROXY_PASSWORD`) and/or `settings.proxy.server`. Proxy is fully optional and **all-or-nothing**: `loadProxyCredentials()` returns `null` when unset (run proxy-less) and throws if only partially set, so a half-configured proxy can never silently leak the real IP. Use a **Singapore, sticky** session ‚Äî the app shares one `BrowserContext` across all workers, so one stable IP for the whole run is correct (rotating mid-session would break the logged-in cookies). **Pin the SAME exit IP across runs**, not just within a run: Lazada/Alibaba bind a logged-in session to its IP, so a new Singapore exit IP each run reads as "logging in from a new location" and is a top trigger for the post-login MF01 security-verification ‚Äî and the growing set of distinct IPs is what makes it escalate. Use a Decodo sticky session with a **fixed session id** in the username (longest TTL) or a **static/ISP residential** IP, and keep it Singapore so it stays coherent with the hardcoded `Asia/Singapore` timezone + `en-SG` locale (a timezone/IP-country mismatch is itself a fraud signal). To check whether your IP is stable, point a config item URL at an IP-echo (`https://ip.decodo.com`) and compare the egress across two separate runs. On top of the proxy, `index.ts` sends UA-consistent client-hint headers and applies the `stealth.ts` fingerprint-masking init script (when `settings.stealth`, default true; see the `stealth.ts` module role). `settings.blockHeavyAssets` (default `true`) aborts image/media/font requests to cut proxy bandwidth ~10√ó and speed each poll; it is auto-disabled when capturing DOM snapshots (`--debug-dom`/`debugSnapshots`) so screenshots stay meaningful. Note: a managed unblocker like Oxylabs Web Unblocker is intentionally **not** integrated ‚Äî it returns stateless HTML and cannot drive the authenticated, interactive PayNow checkout.

## Architecture

`src/index.ts::main` is the orchestrator. There are two monitor modes, chosen by `settings.monitorMode`:

**`per-item`** ‚Äî a strict three-stage flow per item, with safety gates between each stage:

```
config + env  ‚îÄ‚îÄ‚ñ∫  auth (load/restore session, login if needed)
                       ‚îÇ
                       ‚ñº
   For each item, in parallel (Promise.allSettled):
       monitor.waitForStock ‚îÄ‚îÄ‚ñ∫ decision.shouldProceed ‚îÄ‚îÄ‚ñ∫ checkout
              (polls)              (pure function)         (Buy Now ‚Üí PayNow QR)
```

**`wishlist`** (default) ‚Äî one watcher polls the account wishlist and fans out restock signals to N buyers (one full page load per cycle regardless of item count ‚Üí far lower anti-bot footprint; stock is read from embedded JSON at domcontentloaded, so detection pays no render-settle tax). Buyer pages are pre-opened and warm. Use `per-item` only as the fallback when an item can't live on the account wishlist or its title/id won't match:

```
config + env ‚îÄ‚îÄ‚ñ∫ auth ‚îÄ‚îÄ‚ñ∫ one watcher page + N buyer pages
   wishlist-monitor.watchWishlist  ‚îÄ‚îÄfire(productId)‚îÄ‚îÄ‚ñ∫  RestockRegistry gate
       (classifies every item per poll)                         ‚îÇ
                                                                 ‚ñº
                                          runBuyerWorker awaits gate, then
                                          reload PDP ‚îÄ‚îÄ‚ñ∫ checkout (fast retry profile)
```

In wishlist mode the watcher is a single point of failure: on `login_required` it halts the run (aborts the controller) so all buyers exit for re-auth.

### Module roles

- **`index.ts`** ‚Äî Calls `dotenv.config()` to load `.env`, validates config, launches Chromium via **`launchPersistentContext(settings.userDataDir)`** (a real on-disk profile ‚Äî **not** an ephemeral `newContext`; this is what carries cookies + localStorage + IndexedDB + a stable canvas/WebGL fingerprint across runs, so Lazada/Alibaba device-trust tokens survive and a once-verified device stays verified, cutting how often the post-login MF01 security-verification fires), with anti-automation flags, the optional residential proxy, and UA-consistent client-hint headers all on the persistent-context options. There is **no separate `Browser` object** ‚Äî the context is returned directly. Applies the stealth init script (when `settings.stealth`), registers the `blockHeavyAssets` route handler, and loads the cookie backup (`loadSession`) on top of the profile. A 60s `setInterval` backs cookies up to `sessionFile` as insurance against an unclean kill. Branches on `settings.monitorMode`: `per-item` spawns one `runItemWorker` per item (passes `settings.fastCheckout` to `checkout`); `wishlist` opens one watcher page + N pre-opened buyer pages, builds a `RestockRegistry`, runs `watchWishlist`, and spawns one `runBuyerWorker` per item (idles on its gate, then reloads its warm PDP and runs `checkout` with a fast retry profile). SIGINT/SIGTERM (and normal completion) trigger an **idempotent** graceful shutdown that saves the cookie backup and `context.close()`s ‚Äî closing the persistent context is what flushes the profile (localStorage/IndexedDB) to disk for the next run.
- **`wishlist-monitor.ts`** ‚Äî Wishlist watcher. `watchWishlist` polls `settings.wishlistUrl` and classifies **every** tracked item per poll from the **JSON Lazada embeds in the served HTML** (`decision.parseWishlistStock`: `lightItemDetailDTO` = all wishlist items, `outOfStock` = the OOS subset). Config item ‚Üí wishlist entry matching is **title-first** (`decision.matchWishlistItem`: exact normalized title, then unique title-substring, then URL id via `extractProductId`) because config URLs can be non-canonical ‚Äî but classifications and the registry/gates stay keyed by the **config-URL product id**. No DOM scraping ‚Äî robust to UI/CSS changes and needs no render settle (the data is present at domcontentloaded). `fire`s the item's gate on an out_of_stock‚Üíin_stock transition (pure `isRestockTransition`). Mirrors `monitor.ts` challenge-survival; halts loudly on `login_required`. **Pauses while any buy is in flight** (`isPaused` callback wired to a shared `activeBuys` counter in `index.ts`): the buy is the session's peak request rate, and stacking wishlist reloads on top primes anti-bot punishment ‚Äî fired gates are latched, so pausing loses nothing for the item being bought. `verify-selectors` reports the parsed wishlist payload (item count + per-config-item match), not selectors. When a match fails (`item_not_found_on_wishlist`), run `npm run list-wishlist` to see the wishlist's actual ids/titles/stock and fix the config.
- **`restock-signal.ts`** ‚Äî `RestockRegistry` / `RestockGate`: the producer/consumer link. Latched, edge-triggered, re-armable gates keyed by product id. The watcher fires gates; buyers `await waitForRestock(signal)`. Impure (holds promises) so it lives outside `decision.ts`. `abortAll()` + controller abort wake idle buyers on shutdown.
- **`config.ts`** ‚Äî Loads + validates `config.json`. `loadCredentials()` is the only path to env-var credentials and throws if missing; `loadProxyCredentials()` is the equivalent for the optional proxy (returns `null` when unset, throws on partial config). Strict key allow-listing ‚Äî unknown keys throw (including inside the nested `settings.proxy`, which also rejects `username`/`password`). `dryRun` and `blockHeavyAssets` default to `true`; `monitorMode` defaults to `"wishlist"` (in wishlist mode validation additionally requires every item URL to yield a product id).
- **`auth.ts`** ‚Äî Session cookies persisted to disk with mode `0o600` (a backup layer on top of the persistent profile, which is the authoritative session store ‚Äî see `index.ts`). `detectChallenge` checks URL patterns (pure `decision.isChallengeUrl`: `/baxia/`, `/block`, `/robot`, `captcha`, `/cdn-cgi/challenge-platform/`, `/awswaf/`, `/sec/`, plus the Alibaba x5sec punish shape `_____tmd_____`/`/punish`/`x5secdata` seen live) and DOM markers (captcha frames, slider, rate-limit pages). Any challenge throws `ChallengeDetectedError` ‚Äî the script never attempts to solve CAPTCHAs, MFA, or rate limits. Lazada moved auth to `member.lazada.sg` ‚Äî the login page is `settings.loginUrl` (default `https://member.lazada.sg/user/login`; the old `www.lazada.sg/customer/account/login/` 404s, which is what silently broke login). `isLoggedIn` uses **positive + negative** detection (`login.loggedInIndicator` = `#myAccountTrigger`/account section vs `login.loginLink` = `#anonLogin`) and **fails toward logged-out** when neither matches ‚Äî but only after **one `recheckLoginMarkers` retry** (1s settle + re-resolve) on the genuinely-ambiguous case, so a transient slow render doesn't force an unnecessary fresh login (each fresh login is an MF01-risk event); the retry still requires a *positive* logged-in marker to return true, so do not regress to inferring "logged in" purely from the absence of a login link (that false-positive skipped login on a dead session). **The login itself is paced like a human** (`fillInputHumanlike` in `browser-actions.ts`: per-key typing delay + randomized `humanPause`s between fields/before submit) because a sub-second fill+submit is itself a bot tell that raises the post-login MF01 odds; credential values are still env-only, never logged, and the password ref is overwritten right after typing. `runAuthDebug` (CLI `--auth-debug`) clears cookies (keeps the profile's device-trust localStorage, so the observed login is a verified-device one) and steps through the login visibly, snapshotting each step to `data/debug/auth/`.
- **`monitor.ts`** ‚Äî `waitForStock` polls a product page at `checkIntervalMs` with an interruptible sleep (checks `AbortSignal` every 500ms). **Stock detection is buy-button-first and deliberately minimal** (`waitForBuySignal`/`classifyBuySignal`): each check waits *only* for an **enabled Buy Now button** (capped at `pollSettleMs`) and returns `in_stock` the instant it is clickable ‚Äî no price read, no exact OOS reason, not even the challenge check on this hot path, because every extra read before the click risks losing the sub-2s restock window and checkout's fail-closed PayNow guardrail is the real safety net. The wait also short-circuits to `out_of_stock` the moment a definitive OOS marker (incl. a disabled Buy Now) appears, **or when the price anchor is visible with no enabled Buy Now** (hydration gate: the page rendered, it just isn't buyable ‚Äî price is a render signal only, never read as a value), so an OOS poll doesn't burn the full settle even when Lazada's OOS markup matches no candidate; only a slow-hydrating/blank/challenge page spends it all. **Price is informational only** ‚Äî read best-effort on the `out_of_stock` branch for the audit log, `null` on the `in_stock` fast path; it never gates the buy (`shouldProceed` relies on `isPriceAcceptable(null,‚Ä¶)===true`). Because status is decided by the button (not by whether the price rendered), the old spurious `unknown`-on-slow-render is gone ‚Äî `unknown` now means only a navigation failure. Defaults are intentionally aggressive; raise `pollSettleMs`/`checkIntervalMs` for a calmer watch (and `pollSettleMs` must exceed proxied buy-box hydration time, else a real restock is missed). On `login_required` it returns immediately. On `anti_bot` (detected on the non-`in_stock` branch): if `settings.surviveChallenges` is true (default) it backs off via `computeChallengeBackoff` (exponential from `challengeBackoffBaseMs`, capped at `challengeBackoffMaxMs`) and **resumes** monitoring so a single challenge doesn't end the 2-hour watch ‚Äî a circuit breaker (`maxConsecutiveChallenges`) stops endless hammering and returns `anti_bot`. With `surviveChallenges: false` it returns `anti_bot` immediately (legacy fail-closed).
- **`decision.ts`** ‚Äî **Pure functions only.** No I/O, no Playwright, no fs, no console. `shouldProceed`, `isPriceAcceptable`, `computeBackoff`, `computeChallengeBackoff`, `isChallengeUrl` (the anti-bot URL pattern list, incl. the x5sec punish shape), `extractProductId` (parses both Lazada URL shapes: `‚Ä¶-i<digits>.html` and id-first `‚Ä¶/i<digits>-s<sku>.html`), `isRestockTransition`, `parseWishlistStock`/`classifyFromWishlistState` (read stock + decoded titles from the wishlist's embedded `lightItemDetailDTO`/`outOfStock` JSON with a string-aware array scanner), `matchWishlistItem`/`normalizeTitle`/`decodeJsonString` (title-first config‚Üíwishlist matching), `formatOrderSummary`. Keep this discipline ‚Äî these are the unit-test core. Note: `isPriceAcceptable(null, ‚Ä¶)` returns `true` (unknown price is not a skip reason ‚Äî the user controls payment via the PayNow QR).
- **`checkout.ts`** ‚Äî **Dry-run guard is the first check** (before any navigation). **Fast path:** when called with `alreadyOnProductPage` (driven by `settings.fastCheckout`), the *first* attempt skips both the rate-limiter wait and the re-navigation and acts on the live in-stock page the monitor just left ‚Äî this is the key change that makes the sub-3s window winnable. Retries (and all attempts when fast path is off) re-navigate with a rate-limit slot as before. Clicks **Buy Now** only ‚Äî no Add to Cart fallback ‚Äî but **waits (bounded, `BUY_NOW_WAIT_MS`) for the button via `waitForSelectorSet`** rather than sampling the instant page state: Lazada PDP/checkout pages hydrate after domcontentloaded, so instant checks always miss (this exact bug produced 5√ó `buy_now_unavailable` at ~110ms each on a live run). The same waiting applies to the checkout page's Place Order CTA, the PayNow option scan, and confirmation detection. **Checkout-page arrival is element-gated, not URL-gated** ‚Äî "we are on the checkout page" := the Place Order CTA is visible (`CHECKOUT_READY_WAIT_MS=15s`). The old `waitForUrl(/checkout path/)` gate timed out on every live run because Lazada SG checkout lives on the `checkout.lazada.sg` **host** (pathname `/`), and its `load` event may never fire. **Checkout-ready and PayNow selection are one OVERLAPPED loop (`waitForCheckoutReadyAndPayNow`), not sequential:** it drives PayNow selection *while* the page hydrates, so the payment work hides inside the (irreducible, ~1.6s) Buy Now‚Üícheckout transition instead of adding after it ‚Äî this is the change that took a live buy from ~3.4s toward ~2s. It exits only when **both** the Place Order CTA is visible **and** PayNow is positively confirmed (so the fail-closed guardrail can't be bypassed); the CTA's 15s deadline still throws `SelectorNotFoundError` (retryable) and PayNow's `PAYMENT_OPTIONS_WAIT_MS=8s` deadline (measured from CTA-seen) still returns the non-retryable abort. **The loops poll at `CHECKOUT_POLL_MS=75` (was 250)** ‚Äî faster polling is pure client-side CPU (DOM `isVisible`/URL reads, never a Lazada request), so it trims latency with no added anti-bot footprint. **The waits stay challenge-aware:** each pass runs `detectChallenge` (throttled to ~`CHALLENGE_CHECK_MS=300` so the 75ms poll doesn't multiply checks), so an anti-bot punish redirect throws `ChallengeDetectedError` immediately (clean halt + `checkout-challenge-*` snapshot) instead of masquerading as a selector timeout and re-clicking Buy Now into a flagged session ‚Äî exactly what a live x5sec punish produced. **Every failed attempt** logs the page URL and writes an always-on `checkout-failed-<item>-attempt<N>` snapshot (the terminal snapshot only preserves the last attempt's page; the first attempt is usually the informative one). On terminal failure (all retries exhausted) it additionally writes the full forensic snapshot (screenshot + HTML + `describeProductSelectors` report) to `<dataDir>/debug/` ‚Äî none of this is gated by `debugSnapshots`, because that page state is the only evidence of why the buy failed. **PayNow is fail-closed (dev-phase guardrail):** the checkout page offers a saved card next to PayNow, so the overlapped loop must positively confirm PayNow as the *selected* method (via `paymentSelectedIndicator` text, after optionally clicking the PayNow row ‚Äî at most once/sec, `isPayNowConfirmed` is the proof it landed) before Place Order is ever clicked. If it can't confirm within `PAYMENT_OPTIONS_WAIT_MS`, the loop returns `paynow-unconfirmed` and `attemptCheckout` **aborts without placing the order**, logs `paynow_not_confirmed`, writes an always-on `paynow-not-confirmed-*` snapshot, and returns a sentinel error that the retry loop treats as **non-retryable** (re-running Buy Now can't fix a selector mismatch and would stack abandoned checkouts). **The PayNow QR page counts as success:** `waitForConfirmation` accepts *either* the classic order-success page (extracts order number) *or* the PayNow QR / cashier page (`confirmation.paymentPending` markers or a navigation to a `cashier`/`pay` URL) ‚Üí success with `orderNumber: null`, logs `paynow_qr_displayed`. That QR is the user's scan-to-pay handoff: the buyer worker returns (terminates) and the headed browser stays open on the QR while other workers keep running. Both success shapes capture an always-on `order-confirmed-*` snapshot ‚Äî now **fire-and-forget** via `CheckoutResult.pendingSnapshot`: success returns the instant the QR/order page is detected, so the screenshot+HTML write no longer inflates `place_order_to_confirm` or the real time-to-QR; the workers drain `pendingSnapshot` before any `page.close()` (monitor mode closes the page on return) so the capture isn't truncated. Retries with exponential backoff; the profile defaults to `retryBackoffBaseMs/retryBackoffMaxMs/maxRetries` but an optional `RetryProfile` arg overrides it (wishlist buyers pass a fast sub-second profile: `buyRetryBaseMs/buyRetryMaxMs/buyMaxRetries`). The retry loop already covers transient OOS (Buy Now not visible) ‚Äî that is the "traffic vs genuine sellout" retry. `ChallengeDetectedError` is re-thrown so `index.ts` shuts down.
- **`payment-approval.ts`** ‚Äî Human approval gate (stdin readline or Telegram inline keyboard). **Not currently called from `checkout.ts`** ‚Äî the PayNow QR serves as the payment confirmation step. The module is retained for reference if the gate is needed again.
- **`browser-actions.ts`** ‚Äî Thin Playwright wrappers (`navigateTo`, `clickElement`, `fillInput`, `extractText`, `isVisible`, `waitForUrl`). They **only accept `SelectorSet` objects, never raw CSS strings** ‚Äî this is enforced by type. `navigateTo` retries 3√ó with backoff.
- **`selectors.ts`** ‚Äî The single source of truth for every DOM selector in the app. `SELECTORS` is structured by page (`login`, `antiBot`, `product`, `cart`, `checkout`, `confirmation`). Each `SelectorSet` has `candidates` (tried in order, first visible wins), `description`, and `required` (true ‚áí throw `SelectorNotFoundError`; false ‚áí return null). Text-based `:has-text()` candidates are listed last as resilient fallbacks. **Two resolvers:** `resolveSelector` is an *instant* single pass ‚Äî Playwright's `isVisible` ignores its timeout option, so the `timeoutMs` arg never waited and never will; `waitForSelectorSet(page, set, timeoutMs)` is the waiting counterpart (re-polls until visible or deadline, returns the moment a candidate appears). Use the waiting one anywhere the element may still be hydrating (checkout does); the monitor builds its own bounded wait loop (`waitForBuySignal`) on top of the instant resolver to catch the Buy Now button the moment it hydrates. (Wishlist mode does **not** use selectors ‚Äî it parses embedded JSON; see `wishlist-monitor.ts`.) **Selectors are unverified against live Lazada pages ‚Äî run `npm run verify-selectors` after any Lazada UI change.**
- **`diagnostics.ts`** ‚Äî DOM-capture helpers for debugging selector issues. `captureDomSnapshot` saves a full-page screenshot + HTML to `<dataDir>/debug/`. `describeProductSelectors` logs each `SELECTORS.product` set with `{ winner, matchCount, firstText, firstVisible, firstEnabled }` ‚Äî the `matchCount` shows whether a selector resolves once (real buy-box) or many times (carousels). Activated by `--debug-dom` or `"debugSnapshots": true` in config; no-ops otherwise.
- **`health.ts`** ‚Äî Optional local HTTP metrics server. `startHealthServer(port, getStatus)` binds to `127.0.0.1` only (never `0.0.0.0`) using Node's built-in `http` module. Responds to any request with a JSON `RuntimeStatus` snapshot (per-item check counts, last stock status, checkout/purchase tallies). Enabled only when `settings.healthPort > 0`; disabled by default (`healthPort: 0`).
- **`rate-limiter.ts`** ‚Äî Per-domain `acquire(domain)` enforces `minIntervalMs` since last request plus random jitter up to `maxJitterMs`. Called before every page navigation in `monitor` and on `checkout` retries ‚Äî but **deliberately skipped on the fast-checkout first attempt**, where the latency would lose the drop.
- **`stealth.ts`** ‚Äî `applyStealth(context)` adds a browser-context init script (run before page scripts on every navigation) that masks common automation fingerprints: `navigator.webdriver`, empty `plugins`/`mimeTypes`, `languages`, `window.chrome`, the notifications permission query, WebGL vendor/renderer, and hardware profile. This is **avoidance, not bypass** ‚Äî it lowers how often a challenge fires; it never solves a CAPTCHA. Enabled by `settings.stealth` (default true). Shipped as a string because the project has no DOM lib.
- **`logger.ts`** ‚Äî Append-only JSONL audit log at `logs/audit-YYYY-MM-DD.log` plus colored stdout/stderr. Never truncates. Logging errors are swallowed so the main process can never crash from log I/O.
- **`timing.ts`** ‚Äî `PhaseTimer` (injectable clock) records wall-clock durations between named marks (`mark` = delta since the previous mark; `markAt(name, at)` = delta up to an explicit absolute timestamp, for **overlapping** phases ‚Äî pass non-decreasing `at`s, clamp at the call site). Used to measure the buy path: `checkout.ts` logs `checkout_timing` (phase breakdown: nav/locate_buy_now/buy_now_to_checkout/select_payment/place_order_to_confirm), and the workers log `instock_to_confirm` (headline detection‚Üíconfirm `durationMs`; the wishlist buyer also splits out `reloadMs`). Since checkout-ready and PayNow now overlap, `buy_now_to_checkout` is marked at CTA-seen and `select_payment` at PayNow-confirmed via `markAt` ‚Äî so **`select_payment‚âà0` is the intended signal that payment fully hid under the hydration window**, and a non-zero value means payment was the residual bottleneck. **Timing only populates on a real purchase** ‚Äî `checkout` short-circuits at the dry-run guard, so `dryRun:false` is required to see buy-phase timings.

### Cross-cutting invariants

- **Credentials never touch `config.json` or logs.** `config.ts` rejects credential keys at the top level and inside `settings.proxy`. Proxy username/password are env-only (`PROXY_USERNAME`/`PROXY_PASSWORD`); `index.ts` logs only the proxy server, never the creds. After login, `auth.ts` overwrites the in-memory password string immediately.
- **Dry-run defaults to true** at both `DEFAULTS` (config.ts) and the example config. Live purchases require explicitly setting `"dryRun": false` AND not passing `--dry-run`. CLI flag always wins (force-enables dry-run).
- **Anti-bot handling ‚Äî tuned for a 2-hour continuous watch, not strictly fail-closed.** During monitoring, a challenge backs off and resumes (see `monitor.ts`) so the watch survives, bounded by `maxConsecutiveChallenges`. During **login** and **checkout** it is still fail-closed: `ChallengeDetectedError` propagates and halts. The script never *solves* a CAPTCHA/MFA ‚Äî survival means waiting it out and retrying, plus stealth to avoid tripping it. Set `surviveChallenges: false` to restore the old halt-on-first-challenge behaviour.
- **Selector additions go through the abstraction.** Don't inline `page.locator("‚Ä¶")` calls outside `selectors.ts`/`browser-actions.ts`. Add a new `SelectorSet` to the relevant group in `SELECTORS` with multiple candidates and pick the right `required` flag.
- **Payment is always PayNow.** `checkout.ts` hardcodes the method to `"paynow"` regardless of config. Lazada displays a QR code after Place Order is clicked; the user scans it to transfer funds. This is the final human confirmation step.
- Tests run against pure logic (`decision`, `rate-limiter`, `config`). There are no Playwright/integration tests ‚Äî exercise UI changes manually via `verify-selectors` and headed `dev:dry-run`.

## Runtime artifacts (git-ignored)

- `config.json` ‚Äî your real config
- `.env` ‚Äî your credentials
- `data/` ‚Äî runtime artifacts root (configurable via `settings.dataDir`, default `"data"`)
  - `data/profile/` ‚Äî **persistent Chromium profile** (`settings.userDataDir`); the authoritative session + device-trust store (cookies/localStorage/IndexedDB/stable fingerprint). Deleting it forces a fresh login (and likely a fresh MF01). Only one run may use it at a time ‚Äî Chromium locks the dir.
  - `data/session.json` ‚Äî cookie backup (mode 0600), loaded on top of the profile and flushed every 60s + on shutdown
  - `data/logs/audit-YYYY-MM-DD.log` ‚Äî JSONL audit trail
  - `data/debug/<item>-<timestamp>.png` / `.html` ‚Äî DOM snapshots (only when `--debug-dom` is active)
- `dist/` ‚Äî tsc output

## Development workflow

For every task:

1. Sync the feature branch with the latest `main`: `git fetch origin main && git merge origin/main`
2. Complete the task
3. Update all relevant documentation (CLAUDE.md, inline comments, config.example.json)
4. Sync with `main` again and resolve any merge conflicts before pushing
5. Create a pull request
